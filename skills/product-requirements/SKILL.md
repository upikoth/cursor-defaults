# Требования к продукту (tracking)

## Когда применять

- Пользователь описывает новую фичу или изменение.
- Нужно уточнить или зафиксировать требования до кода.
- Нужно обновить существующий файл в `docs/requirements/features/`.

## Источник правды

Полный текст вопросов по блокам, запреты и встроенный шаблон — в `.cursor/rules/requirements.mdc`. Шаблон для копирования — `docs/requirements/features/template.md`. При расхождении следовать **requirements.mdc**.

## Чеклист процесса

1. **Уточнение.** Задать вопросы по всем неочевидным пунктам (акторы, поведение, данные, API, UI, scope). Не пропускать шаг из‑за «простоты» задачи.
2. **Документ.** Заполнить файл `docs/requirements/features/{feature-name}.md` (kebab-case) по шаблону. Acceptance criteria — в формате Given / When / Then, без пустых обязательных блоков.
3. **Подтверждение.** Показать пользователю итог и спросить явно: всё верно, можно ли приступать к реализации. Код **только после** ответа вроде «можно приступать», «реализуй», «всё верно».
4. **Реализация.** Опираться на acceptance criteria; порядок: backend → frontend (если не оговорено иначе); тесты под критерии; после работы статус в файле: `Draft` → `Implemented`.

## Запрещено

- Писать код до заполненного файла требований и явного подтверждения.
- Реализовывать то, что помечено как вне скоупа в том же документе.
