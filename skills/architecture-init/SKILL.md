# Первичная архитектура (architecture-init)

## Когда применять

- Новый продукт или сервис: папка `docs/architecture/` **пуста** или нет `system-overview.md` / принятых ADR.
- Пользователь просит **спроектировать архитектуру**, **создать ADR**, **описать компоненты** после утверждения требований.
- Нужен **первый** обзор системы и набор решений перед кодом.

**Не применять** для изменения уже принятой архитектуры — см. раздел «Поддержка и эволюция» в `.cursor/rules/architecture.mdc` (новый ADR, supersede, changelog).

**Не подменять** сбор FR или NFR — навыки `functional-requirements` и `non-functional-requirements`.

## Источник правды

- Структура и поддержка: **`.cursor/rules/architecture.mdc`**
- Эталон ADR: **`docs/architecture/adr/template.md`**
- Требования: утверждённые файлы в `docs/requirements/functional/`, `docs/requirements/non-functional/`
- Стек по умолчанию продукта: **`.cursor/rules/project.mdc`** (Go, Vue3, PostgreSQL), если человек не указал иное

## Предусловия (обязательно проверить)

1. В `functional/` есть документ(ы) со **`status: approved`** и заполненными **FR-{AREA}-{NNN}** для описываемого MVP или области.
2. Если архитектурный выбор зависит от качества (PERF, SEC, OBS, DATA, …) — в `non-functional/` есть **`status: approved`** NFR или человек явно согласился отложить NFR с фиксацией риска в ADR.
3. Человек инициировал создание архитектуры или согласился на сессию после требований.

Если предусловия не выполнены — **остановиться**, предложить сначала завершить FR/NFR (соответствующие навыки).

## Результат сессии (артефакты)

После прохождения чеклиста в репозитории появляются:

```
docs/architecture/
├── README.md
├── system-overview.md        # status: draft до утверждения
├── changelog.md              # первая запись о создании
└── adr/
    ├── README.md             # таблица ADR
    ├── ADR-001-….md          # status: proposed → accepted после «утверждаю»
    ├── ADR-002-….md
    └── …
```

Минимальный набор ADR для типичного веб-продукта (адаптировать под FR/NFR):

| ADR | Тема (пример) |
| --- | ------------- |
| ADR-001 | Стек, границы сервисов, локальный/деплой сценарий |
| ADR-002 | Модель данных и хранение |
| ADR-003 | API-контракт (REST/OpenAPI), привязка к FR |
| ADR-004+ | OBS, auth, интеграции — по NFR и FR |

Имена сервисов — **kebab-case**, роль в суффиксе: `*-api`, `*-web` (см. `project.mdc`).

## Чеклист процесса

### Шаг 1 — Контекст и граница

- Прочитать утверждённые **FR-*** и релевантные **NFR-***.
- Зафиксировать границу системы (что внутри диаграммы, что снаружи).
- Список акторов и каналов из FR; ограничения из NFR (локальный периметр, лимиты, логи и т.д.).

### Шаг 2 — Вопросный лист (только недостающее)

Задай блоком вопросы, без которых нельзя принять решения; пропускай уже зафиксированное в FR/NFR:

1. **Сервисы:** сколько приложений (API, SPA, воркеры); имена каталогов.
2. **Данные:** сущности, связи, лимиты из FR/NFR; миграции PostgreSQL.
3. **API:** REST vs иное; версионирование; ошибки на уровне контракта (детали — в OpenAPI у сервиса).
4. **Интеграции:** внешние системы из FR (если есть).
5. **Эксплуатация:** локальный запуск, конфиг через env, логи, request-id (NFR-OBS).
6. **Открытые альтернативы:** что сознательно **не** выбираем в MVP.

Не выбирать библиотеки без необходимости; достаточно уровня «Go + chi/echo/…» только если человек просит или без этого нельзя зафиксировать ADR.

### Шаг 3 — Черновики документов

1. Создать **`docs/architecture/README.md`** — назначение, ссылка на [[system-overview]], как читать ADR и changelog.
2. Создать **`system-overview.md`**:
   - Mermaid C4 Container;
   - таблицы компонентов и трассировки FR/NFR;
   - wikilinks на каждый ADR.
3. Для каждого решения — **`adr/ADR-{NNN}-<slug>.md`** по **`adr/template.md`** (`status: proposed`, заполнить `traces.fr` / `traces.nfr`).
4. **`adr/README.md`** — таблица: id, title, status, date, supersedes/superseded_by.
5. **`changelog.md`** — запись о первичном создании набора ADR (дата, перечень ADR, «начальная архитектура MVP»).

Если папки или эталоны отсутствуют — создать по структуре из `architecture.mdc`.

### Шаг 4 — Согласование с человеком

Показать:

- Mermaid из `system-overview.md` (или краткое описание);
- таблицу **ADR-001 … ADR-NNN** с однострочным решением;
- явные стыковки и **противоречия** с FR/NFR (если есть);
- открытые вопросы.

Запросить **явное утверждение** («утверждаю», правки списком). До утверждения:

- ADR остаются `proposed`, overview — `draft`;
- **не начинать** код сервисов и не генерировать OpenAPI в репозиториях кода.

### Шаг 5 — Фиксация утверждения

После «утверждаю»:

- ADR → `status: accepted`, актуальная `date`;
- `system-overview.md` → `status: approved`, инкремент `version` (например `1.0`);
- обновить `adr/README.md` и при необходимости `changelog.md`.

Сообщить следующий шаг: реализация по `backend.mdc` / `frontend.mdc` **только** по явному поручению.

## Трассируемость

В каждом ADR в **`traces.fr`** и **`traces.nfr`** — только id, реально затронутые этим решением. В overview — сводная таблица «компонент → FR/NFR → ADR».

OpenAPI в ADR-003 (или аналоге): указать **путь будущего файла** `<имя-api>/docs/openapi.yaml` и перечень операций, покрывающих **FR-***, без полной спецификации в `docs/architecture/`.

## Запрещено

- Создавать архитектуру без утверждённых FR для описываемого поведения.
- Писать код приложений в той же сессии без отдельного явного поручения после утверждения архитектуры.
- Один файл вместо overview + ADR + changelog.
- Редактировать принятые ADR при последующих изменениях — только новые ADR и supersede (не относится к первичному init, но важно не смешивать процессы).
- Добавлять зависимости в код — см. `project.mdc`; в ADR допустимы технологии уровня стека проекта.
