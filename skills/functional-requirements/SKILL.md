# Требования к продукту

## Когда применять

- Пользователь описывает новый продукт, доменную область или изменение поведения — нужно вести или править записи **`FR-{AREA}-{NNN}`** в `docs/requirements/functional/`.
- Нужно уточнить требования до архитектуры или кода или свериться с уже утверждёнными документами.
- Нужно добавить или обновить документы в `docs/requirements/non-functional/`.

## Источник правды

Полные правила, вопросные листы и запреты — в `.cursor/rules/requirements.mdc`. Эталон для области — **`docs/requirements/functional/template.md`**.

## Чеклист процесса (функциональный корпус)

1. Зафиксировать область, код **AREA** и границы по разделу «Обязательный процесс для агента» в `requirements.mdc`.
2. Пройти вопросный лист по блокам шаблона (контекст, глоссарий, акторы, FR-\* с проверяемыми критериями, интеграции, связь с НФТ при наличии).
3. Заполнить или изменить файл `docs/requirements/functional/<область-kebab>.md` по образцу `template.md`.
4. Показать итог и получить явное подтверждение человека перед счётом изменений надёжной опорой для кода или ADR.

## Запрещено

- Кодировать поведение без утверждённых связанных **FR** и явного человеческого согласия на порцию работ (если задача затрагивает продукт, а не заранее оговорённый чистый рефакторинг).
- Вводить новые возможности только в коде без обновления `functional/` и утверждения.
